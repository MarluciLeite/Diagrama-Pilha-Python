Resumo

Este PR adiciona um esqueleto completo de projeto Python com foco em facilidade de desenvolvimento e qualidade de código. Objetivo: fornecer uma base mínima para implementar e testar uma estrutura de pilha (`stack`) em Python, com ferramentas e CI já configuradas.

O que foi incluído

- Estrutura de pacote em `src/pilha` com implementação `Stack` (`stack.py`) e `__init__.py`.
- Testes básicos em `tests/test_stack.py` usando `pytest`.
- Arquivos de configuração de desenvolvimento:
  - `.gitignore`, `requirements.txt`, `requirements-dev.txt`.
  - `pyproject.toml` com configurações para `black`, `ruff` e `mypy`.
  - `.pre-commit-config.yaml` com hooks para `black` e `ruff`.
- Devcontainer para VS Code:
  - `.devcontainer/Dockerfile` e `.devcontainer/devcontainer.json` (imagem Python 3.12-slim; instala dependências e ferramentas durante o build).
- CI (GitHub Actions): `.github/workflows/ci.yml` que executa `ruff`, `black --check`, `mypy` e `pytest` (com `pytest-cov` para gerar `coverage.xml` e enviar para Codecov).
- Badges no `README.md` para CI e coverage.

O que eu validei localmente

- Criei e ativei um ambiente virtual (`.venv`).
- Instalei dependências de runtime e dev: `pip install -r requirements.txt && pip install -r requirements-dev.txt`.
- Rodei `ruff check src tests` — sem erros.
- Rodei `black` (formatei automaticamente um arquivo) e `mypy src` — sem erros.
- Rodei `pytest --cov=src --cov-report=xml` e confirmei que os testes passaram e `coverage.xml` foi gerado.

Pontos para revisão

- Valide se as configurações do devcontainer atendem ao seu fluxo (instalação global durante o build; se preferir instalação no usuário, posso ajustar).
- Verifique o conteúdo do workflow CI (versão do Python, passos de instalação e report de coverage).
- Cheque se as versões das ferramentas em `requirements-dev.txt` são aceitáveis; posso travar versões específicas se desejar builds totalmente reproduzíveis.

Como testar localmente

1. `python3 -m venv .venv` && `source .venv/bin/activate`
2. `pip install -r requirements.txt`
3. `pip install -r requirements-dev.txt`
4. `PYTHONPATH=src pytest --cov=src --cov-report=xml -q`
5. `ruff check src tests`
6. `black --check .`
7. `mypy src`

Notas finais

- O badge do Codecov só mostrará cobertura após o primeiro run do CI que fizer o upload para codecov.io. Para repositórios privados pode ser necessário configurar `CODECOV_TOKEN` como secret.
- Se quiser eu também posso:
  - adicionar um workflow para formatar automaticamente e commitar (ou criar PR) com `black`,
  - adicionar reviewers automaticamente ao PR,
  - ou abrir issues separadas para cada melhoria (pre-commit, pinning de versões, template de PR).
